    safe_identifier("users")?,
    vec![
        FieldDef::new(safe_identifier("name")?, StringField::new(), false, None, false),
        FieldDef::new(safe_identifier("email")?, StringField::new(), false, None, false),
        FieldDef::new(safe_identifier("age")?, IntField::new(), true, None, false),

        // timestamps() returns Vec<FieldDef>
    ],
    Some(safe_identifier("id")?),   // custom primary key
);

// Build a changeset from user input
// Params use temper_core::Map
let params = temper_core::Map::new(&[
    ("name", "Alice"),
    ("email", "alice@example.com"),
    ("age", "30"),
]);
let cs = changeset(users_def.clone(), params)
    .cast(vec![safe_identifier("name")?, safe_identifier("email")?, safe_identifier("age")?])
    .validate_required(vec![safe_identifier("name")?, safe_identifier("email")?])
    .validate_length(safe_identifier("name")?, 1, 100)
    .validate_int(safe_identifier("age")?);

// Check validity
if cs.is_valid() {
    let insert_sql = cs.to_insert_sql().to_string();
    // => INSERT INTO users (name, email, age) VALUES ('Alice', 'alice@example.com', 30)
}

// Update existing record

// => UPDATE users SET name = 'Alice', email = 'alice@example.com', age = 30 WHERE id = 42

  .validate_float(safe_identifier("score")?)
  .validate_bool(safe_identifier("active")?)
  .validate_inclusion(safe_identifier("role")?, vec!["admin", "user", "guest"])
  .validate_exclusion(safe_identifier("status")?, vec!["banned"])
  .validate_number(safe_identifier("age")?, 0, 150)
  .validate_acceptance(safe_identifier("terms")?)
  .validate_confirmation(safe_identifier("password")?, safe_identifier("password_confirmation")?)
  .validate_contains(safe_identifier("bio")?, "keyword")
  .validate_starts_with(safe_identifier("code")?, "PRE-")
  .validate_ends_with(safe_identifier("email")?, ".com");

// Data manipulation
cs.put_change(safe_identifier("updated_at")?, "2026-03-13".into())

  .delete_change(safe_identifier("age")?);

## Security Model

The Alloy ORM is built on a defense-in-depth approach to SQL injection prevention, using Temper's type system to enforce safety at compile time. The compiled Rust output preserves all safety invariants.

### Defense Layers

**Layer 1: SafeIdentifier (CWE-89 -- table/column names)**

Table and column names must pass through `safe_identifier()`, which validates against `[a-zA-Z_][a-zA-Z0-9_]*`. The function returns `temper_core::Result<SafeIdentifier>` -- invalid identifiers produce an error rather than silently proceeding. The internal `ValidatedIdentifier` struct is not exported.

```rust
safe_identifier("users")?;        // => Ok(ValidatedIdentifier)
safe_identifier("users; DROP")?;  // => Err (rejected)
safe_identifier("table--name")?;  // => Err (rejected)
```

**Layer 2: Typed SqlPart hierarchy (CWE-89 -- value escaping)**

User values never touch `append_safe`. Each type gets its own `SqlPart` implementation with type-appropriate rendering:

| Type | Rendering | Example |
|------|-----------|---------|
| `SqlString` | Quote-wraps with `'` escaping (`'` becomes `''`) | `'O''Brien'` |
| `SqlInt32` / `SqlInt64` | Bare integer via `.to_string()` | `42` |
| `SqlBoolean` | Literal `TRUE` / `FALSE` | `TRUE` |
| `SqlFloat64` | Bare float (NaN/Infinity render as `NULL`) | `3.14` |
| `SqlDate` | Quote-wrapped date string with `'` escaping | `'2026-03-13'` |
| `SqlDefault` | Literal `DEFAULT` keyword | `DEFAULT` |

The `SqlPart` trait is sealed -- adding a new type without updating the rendering logic is a compile error.

**Layer 3: SqlBuilder separation (CWE-89 -- structural safety)**

`SqlBuilder` enforces a clean separation between SQL structure and data:
- `append_safe(str)` -- for SQL keywords and validated identifiers only
- `append_string(val)` -- wraps in `SqlString` (escaped)
- `append_int32(val)` -- wraps in `SqlInt32` (bare integer)

There is no `append_raw` or bypass method.

**Layer 4: Changeset pipeline (CWE-915 -- mass assignment)**

`Changeset` is a sealed trait -- `ChangesetImpl` is not exported, so construction only happens through the `changeset()` factory. The pipeline enforces:
- `cast(allowed_fields)` -- field whitelisting via `Vec<SafeIdentifier>`, prevents mass assignment
- `validate_required(fields)` -- non-nullable enforcement
- `to_insert_sql()` / `to_update_sql(id)` -- independently enforces non-nullable fields
- Raw `params` are never exposed -- only whitelisted `changes` are readable

**Layer 5: Query builder (CWE-89, CWE-400)**

`Query` requires `SafeIdentifier` for table names, column selections, ORDER BY clauses, and JOIN table names. WHERE and ON conditions accept only `SqlFragment` (never raw strings). `safe_to_sql(default_limit)` enforces result set bounds (CWE-400). `UpdateQuery` and `DeleteQuery` bubble on empty WHERE clauses, preventing accidental full-table mutations.

### Per-Phase Security Analysis

**JOIN Feature Security** -- JOIN keywords use a sealed `JoinType` trait with hardcoded strings. Table names require `SafeIdentifier`. ON conditions accept only `SqlFragment`. The `col()` helper produces qualified column references from two `SafeIdentifier` values joined by a hardcoded `.` separator.

**WHERE Clause Enrichment** -- `WhereClause` is sealed (`AndCondition`/`OrCondition`). Convenience methods (`where_null`, `where_in`, `where_between`, `where_like`, `where_ilike`) build fragments internally using only validated identifiers and typed values. Empty IN list produces `1 = 0`. LIKE/ILIKE patterns go through `SqlString` escaping.

**Aggregation** -- All 6 aggregate functions use hardcoded function names via `append_safe`. Field names require `SafeIdentifier`. `select_expr`, `group_by`, `having`/`or_having` reuse existing safe types. `distinct()` selects between two hardcoded strings.

**Set Operations and Subqueries** -- Set operation keywords (`UNION`, `INTERSECT`, `EXCEPT`) are hardcoded literals. Both operands are `Query` objects. Sub-SELECTs are parenthesized. `subquery()` alias requires `SafeIdentifier`. `exists_sql()` wraps in hardcoded `EXISTS (...)`. `where_in_subquery()` combines `SafeIdentifier` field with `Query` subquery.

**Batch UPDATE/DELETE** -- `SetClause.field` requires `SafeIdentifier`. `SetClause.value` uses `SqlPart` (sealed, type-dispatched). `UpdateQuery` and `DeleteQuery` bubble on empty WHERE clauses, preventing accidental full-table mutations.

**Extended Query Features** -- `NullsPosition` and `CrossJoin` use sealed traits with hardcoded keywords. `LockMode` (`ForUpdate`/`ForShare`) uses sealed traits with hardcoded strings.

**Changeset Enrichment** -- All validation methods operate on the `changes` map (string validation only, no SQL generation). Field parameters require `SafeIdentifier`. Values only reach SQL through `to_insert_sql()`/`to_update_sql()` which use type-dispatched escaping. Immutable pattern preserved.

**Schema Enrichment** -- `TableDef.primary_key` uses `SafeIdentifier`. `pk_name()` returns `SafeIdentifier.sql_value` or hardcoded `"id"`. `FieldDef.default_value` uses `SqlPart` (sealed). `SqlDefault` renders hardcoded `"DEFAULT"`. Virtual fields are excluded from SQL entirely.

## Getting Started

### Installation

Copy (or vendor) the `orm/`, `std/`, and `temper-core/` directories into your Rust project. Add `temper-core` as a dependency in your `Cargo.toml`:

```toml
[dependencies]
temper-core = { path = "vendor/temper-core" }
orm = { path = "vendor/orm" }
```

### Minimal Example

```rust
use orm::src::{from, safe_identifier, changeset, TableDef, FieldDef, StringField, IntField, SqlInt32};

fn main() -> temper_core::Result<()> {
    // Define a table
    let todos_table = TableDef::new(
        safe_identifier("todos")?,
        vec![
            FieldDef::new(safe_identifier("title")?, StringField::new(), false, None, false),
            FieldDef::new(safe_identifier("completed")?, IntField::new(), false, None, false),
        ],
        None,
    );

    // SELECT query
    let select_sql = from(safe_identifier("todos")?)
        .order_by(safe_identifier("created_at")?, true)
        .limit(50)
        .to_sql()
        .to_string();
    println!("{}", select_sql);
    // => SELECT * FROM todos ORDER BY created_at ASC LIMIT 50

    // INSERT via changeset
    let params = temper_core::Map::new(&[("title", "Buy milk"), ("completed", "0")]);
    let cs = changeset(todos_table, params)
        .cast(vec![safe_identifier("title")?, safe_identifier("completed")?])
        .validate_required(vec![safe_identifier("title")?]);

    if cs.is_valid() {
        println!("{}", cs.to_insert_sql().to_string());
        // => INSERT INTO todos (title, completed) VALUES ('Buy milk', 0)
    }

    Ok(())
}
```

## Architecture

This repository is updated automatically by CI. The cascade works as follows:

```
alloy (Temper source)
  |
  | temper build -b rust
  v
temper.out/rust/
  |
  | CI publish (push to this repo)
  v
alloy-rust (this repo)
  |
  | notify-app.yml workflow (triggered on push)
  v
alloy-rust-app (demo application)
  |
  | vendors orm/, std/, temper-core/ into vendor/
  v
Application uses the ORM
```

A push to `main` in the [Alloy source repository](https://github.com/notactuallytreyanastasio/alloy) triggers CI, which builds the Temper compiler, compiles the ORM for all 6 backends, runs the test suite, and pushes the compiled Rust output to this repository. The `notify-app.yml` workflow in this repo then automatically vendors the updated ORM into the [alloy-rust-app](https://github.com/notactuallytreyanastasio/alloy-rust-app) demo application.
